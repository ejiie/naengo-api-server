# 기능 명세서 템플릿

> **사용법**
> 1. 이 파일을 복사해 `docs/spec/<도메인>-<기능>.md` 로 저장한다. 예: `docs/spec/recipe-create.md`
> 2. 아래 섹션을 **모두** 채운다. "해당 없음"이면 그렇게 적되 비워두지 않는다.
> 3. 작성이 끝나면 LLM에게 *"이 명세서대로 코드를 생성해줘. 다른 가정은 하지 말고, 모호한 부분이 있으면 먼저 질문해줘."* 라고 전달한다.
> 4. 코드 받은 뒤 수정 사항이 발생하면 `change-log-template.md` 형식으로 별도 기록한다 (이 명세서 자체는 가급적 수정하지 않는다 — 명세서가 계속 바뀌면 이력이 사라진다).

## 1. 목적 (Why)

이 기능을 왜 만드는가? **사용자(또는 관리자)의 어떤 시나리오를 풀려는 것인지** 2~4문장으로.

좋은 예: *"비로그인 사용자가 메인 페이지에서 인기 레시피를 둘러볼 수 있어야 한다. 회원가입 전 서비스 가치를 보여주는 것이 가입 전환의 핵심이라고 판단했기 때문이다."*

나쁜 예: *"레시피 목록 API를 만든다."* ← 이건 What, Why가 빠짐.

## 3. API 스펙

### 3.1 엔드포인트

| Method | Path | 인증 | 설명 |
|---|---|---|---|
| `POST` | `/api/recipes` | 필수 (USER) | 레시피 작성 |

### 3.2 Request

**Path/Query/Header**
```
(없으면 "없음")
```

**Body** — JSON 스키마와 예시 둘 다.
```json
{
  "title": "string, 1~100자, 필수",
  "ingredients": [{"name": "string", "amount": "string"}],
  "fullContent": "string, 본문, 필수"
}
```

예시:
```json
{
  "title": "김치볶음밥",
  "ingredients": [{"name":"김치","amount":"200g"}],
  "fullContent":"1. 팬에 기름을 두른다 ..."
}
```

### 3.3 Response

성공 (HTTP 201):
```json
{
  "success": true,
  "data": {
    "recipeId": 42,
    "status": "PENDING"
  }
}
```

실패 — 발생 가능한 ErrorCode 전부 표로:

| HTTP | code | 언제 |
|---|---|---|
| 400 | `INVALID_INPUT` | 제목 누락 / 길이 초과 |
| 401 | `UNAUTHORIZED` | 토큰 없음/만료 |
| 403 | `USER_BLOCKED` | 차단된 사용자 |

## 4. 비즈니스 규칙 (Business Rules)

번호로 나열. **여기가 가장 중요하다.** 모호한 규칙이 있으면 코드가 모호해진다.

1. 작성 시 `status = 'PENDING'`, `source = 'USER'` 로 고정한다.
2. `author_id` 는 JWT에서 추출한 user_id 를 사용한다 (Body로 받지 않는다).
3. PENDING 상태는 일반 목록 조회에 노출되지 않는다.
4. 같은 사용자가 동일 제목으로 여러 개 작성하는 것은 허용한다 (중복 검사 없음).

## 9. 테스트 케이스 (수용 기준)

체크리스트로. 각 항목이 통과하면 명세서를 만족한 것으로 본다.

- [ ] 정상 작성 → 201, recipeId 반환
- [ ] 토큰 없이 요청 → 401
- [ ] title 누락 → 400 INVALID_INPUT
- [ ] 차단 사용자 → 403 USER_BLOCKED
- [ ] PENDING 레시피는 `GET /api/recipes` 목록에 안 나온다
- [ ] 본인은 `GET /api/users/me/recipes` 에서 PENDING도 본다
