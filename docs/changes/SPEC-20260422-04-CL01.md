## 1. 무엇이 문제였나 (What)

명세 §4-5 는 `session_logs.selected_recipe_id` 의 `ON DELETE` 가 비지정(=NO ACTION) 이라 가정하고, 애플리케이션이 선행 NULL UPDATE 로 우회한다. 그러나 [`api-server-tasks.md §1.5`](../api-server-tasks.md) 에서 **V4 마이그레이션이 V1 의 보정본**으로 도입되기로 결정됐고, V4 의 후보 항목 1번이 정확히 이 FK 에 `ON DELETE SET NULL` 을 부여하는 것이다.

V4 적용 후에도 명세는 그대로 유효한가? — **유효하다**. 다만 가정이 한 단계 약해진다.

---

## 2. 왜 그렇게 됐나 (Why)

V1 작성 시점에는 `session_logs` 의 FK 에 명시적 `ON DELETE` 를 부여하지 않았다. AI 서버 측 contract (이번에 가시화된 <http://43.201.62.254:8000/docs>) 와 대조하며 보정 마이그레이션을 정리하던 중 발견됐다.

---

## 3. 어떻게 바꿨나 (How)

- **변경 파일**: 없음. 본 명세서·구현 코드 모두 그대로.
- **마이그레이션**: `db/migration/V4__*.sql` 에 다음 항목이 들어갈 예정:
  ```sql
  ALTER TABLE session_logs
      DROP CONSTRAINT IF EXISTS session_logs_selected_recipe_id_fkey;
  ALTER TABLE session_logs
      ADD CONSTRAINT session_logs_selected_recipe_id_fkey
      FOREIGN KEY (selected_recipe_id) REFERENCES recipes(recipe_id)
      ON DELETE SET NULL;
  ```
  (실제 제약 이름은 `pg_constraint` 조회 후 확정)
- **테스트**: `docs/db-testing-guide.md §2` 가 V4 적용 후 `confdeltype = 'n'` 을 확인하도록 갱신됨.
