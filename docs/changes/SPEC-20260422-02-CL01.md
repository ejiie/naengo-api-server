## 1. 무엇이 문제였나 (What)

원본 명세 §3.2 의 요청 Body 는
```json
{ "title": "...", "fullContent": "...", "imageUrl": "...",
  "ingredients": [{"name": "...", "amount": "..."}] }
```
이지만, AI 서버 `RecipeResponse` (응답·RAG 검색 결과로 노출되는 형태) 는 11개 필드가 더 풍부하다 (`description`, `ingredients_raw`, `instructions`, `servings`, `cooking_time`, `calories`, `difficulty`, `category`, `tags`, `tips`, `video_url`). 또한 `IngredientItem` 도 `{name, amount, unit, type, note}` 5필드.

즉 사용자가 작성한 레시피를 AI 가 RAG 검색 결과로 다시 내보낼 때 **필수 필드 (`description`, `ingredients_raw`, `servings`, `cooking_time`, `difficulty`) 가 비어있는 상태**가 된다. 사용자 업로드 경로에서도 이 필드들을 받아야 한다.

상세 갭분석: [`docs/spec/ai-server-contract.md §3.1, §4-1`](../spec/ai-server-contract.md).

---

## 2. 왜 그렇게 됐나 (Why)

`SPEC-20260422-02` 작성 시점에 AI 서버 OpenAPI 가 가시화되지 않았고, 원안은 사용자 입력 부담을 최소화하려고 `title + fullContent + ingredients(name/amount)` 만 받게 단순화했다. AI 서버가 실제로 풍부한 메타데이터를 요구하는 게 드러난 것은 2026-05-02 docs 공개 이후.

---

## 3. 어떻게 바꿀 것인가 (How)

본 CL 은 **방향만 정리**. 실 변경은 V4 머지 후 새 PR 에서 진행:

- **V4 머지** 후 `Recipe` 엔티티 / `RecipeCreateRequest` DTO / `RecipeService.create()` / `RecipeResponse` 응답 DTO 를 새 컬럼에 맞춰 확장
- 명세서 본문은 그대로 두고, **신규 명세 `SPEC-2026MMDD-NN: recipe-create v2`** 를 발행해 새 입력 schema 를 정의
  - 결정 사항: 어떤 필드를 필수로 할지 / 입력은 옵셔널이지만 AI 노출은 필요한 필드의 디폴트 처리
- `IngredientItem` 의 `unit`/`type`/`note` 도 입력에 받음 (DB 단 JSONB 라 DDL 변경 없음)

V4 자체에는 schema (DDL) 만 들어가고 입력 검증·DTO 는 별도 PR 로 분리.
